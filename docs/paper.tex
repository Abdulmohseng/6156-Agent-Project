\section*{Synopsis}

\textbf{NeatAgent} is a fully local, privacy-preserving AI agent that autonomously organizes files on a user's filesystem. Given a target folder, the agent reads file contents, semantically classifies each file, generates a structured plan of filesystem operations, presents it to the user for approval, and then executes it — all without sending any data to a cloud service. No subscription, no API key, no internet connection required.

The system is built on a \textbf{PLAN-AND-ACT} architecture, a pattern in which an agent first generates a complete plan before taking any action. The plan is implemented as a LangGraph state machine — a graph of connected nodes (Planner, Executor, Reflector) that communicate through a shared state dictionary. Inference is performed locally using \textbf{Ollama}, a tool that serves quantized open-weight large language models (LLMs) on consumer hardware.

\textbf{Key terms for readers unfamiliar with AI agents:}
\begin{itemize}
    \item \textit{LLM (Large Language Model)}: A neural network trained on large amounts of text that can generate human-like responses, follow instructions, and produce structured output (e.g., JSON).
    \item \textit{AI Agent}: A system that uses an LLM to reason about a task, select actions (called \textit{tools}), execute them, and react to results — operating in a loop until the goal is achieved.
    \item \textit{Hallucination}: When an LLM confidently generates output that is factually wrong or entirely fabricated — for example, inventing filenames that do not exist in the folder being organized.
    \item \textit{Quantized model}: A compressed version of an LLM that trades a small amount of accuracy for dramatically reduced memory and compute requirements, making it runnable on a laptop.
\end{itemize}

The core innovation of NeatAgent is a \textbf{mapping-based planner} that makes hallucination of non-existent files structurally impossible: rather than asking the LLM to generate executable steps (which can drift and invent files), the planner asks only for a \texttt{\{filename $\to$ folder\}} mapping keyed by the exact filenames provided in the prompt. All steps are then generated deterministically in Python. NeatAgent is open-source, runs on macOS, Linux, and Windows, and is launched with a single command: \texttt{python run.py}.

% ─── Abstract ────────────────────────────────────────────────────────────────
\begin{abstract}
Most file organization agents rely on proprietary cloud APIs, raising concerns around privacy, cost, and offline availability. This paper presents NeatAgent, a terminal-based file organization agent that runs entirely on consumer hardware using local LLMs served by Ollama. The agent implements a PLAN-AND-ACT loop with a Reflector node for error recovery and a mapping-based planner that structurally eliminates hallucinated file references — a failure mode we observed producing up to 42 fabricated filenames on a 10-file input with the original step-based planner. We evaluate the system through three research questions: (1) Can local LLMs organize files meaningfully without cloud APIs? (2) How do individual pipeline components contribute to output quality (ablation study)? (3) How does model size affect placement accuracy and speed? Across three models (\texttt{qwen2.5-coder:14b}, \texttt{qwen3:8b}, \texttt{llama3.2:3b}) and two labeled datasets, all models achieve 100\% file coverage. Placement accuracy ranges from 40\% to 60\%, with the larger model outperforming smaller ones. The ablation study shows the mapping-based planner is the single most impactful component, and that content previews improve placement accuracy for files with ambiguous names.
\end{abstract}

% ─── Section 3: Novel / Innovative ───────────────────────────────────────────
\section{Novel and Innovative Aspects}

\begin{enumerate}
    \item \textbf{Mapping-based hallucination prevention.} Prior step-based planners ask the LLM to generate executable action steps, which drifts and invents filenames (Section~\ref{sec:ablation}). NeatAgent's planner instead outputs a \texttt{\{filename $\to$ folder\}} JSON mapping keyed by the exact filenames it was given. Steps are generated by Python code. Any hallucinated key is filtered out before steps are built. This design makes hallucination of non-existent files structurally impossible.

    \item \textbf{Content-aware classification for ambiguous files.} For files with generic names (e.g., \texttt{file4.pdf}), NeatAgent extracts a text preview — using \texttt{pdfplumber} for PDFs, \texttt{python-docx} for DOCX, and an Ollama vision model for images — and injects it into the planner prompt as a content hint. This provides the LLM with signal it cannot derive from the filename alone.

    \item \textbf{Fully local and private.} No cloud API is used at any point. All LLM inference runs on the user's machine via Ollama. Files are never uploaded or transmitted.

    \item \textbf{Local LLM-as-judge evaluation.} The evaluation pipeline uses a local Ollama model as a semantic quality judge, scoring organization quality 1--10 with structured JSON output. This requires no external API key and preserves the local-only design end-to-end.
\end{enumerate}

% ─── Section 5: Research Questions ───────────────────────────────────────────
\section{Research Questions and Answers}

\subsection{RQ1: Can a locally-hosted LLM meaningfully organize files without cloud dependency?}

\textbf{Motivation.} The dominant paradigm for capable AI agents uses frontier cloud models (GPT-4, Claude). The hypothesis is that quantized open-weight models, served locally, can achieve practically useful organization quality without sacrificing privacy or incurring API costs.

\textbf{Methodology.} We ran NeatAgent on the \texttt{model\_test} dataset (10 files with generic names: \texttt{file1.txt}–\texttt{file10.csv}, covering invoices, health reports, gym logs, CSVs, and photos) using \texttt{qwen2.5-coder:14b}. We measured \textit{coverage} (fraction of files moved out of root) and \textit{placement accuracy} (fraction placed in the correct semantic folder per ground truth with alias matching).

\textbf{Results.} The 14B model achieved 100\% coverage and approximately 60\% placement accuracy. Every file was organized into a semantically meaningful folder, and the majority were placed correctly. Representative output: \texttt{file1.txt} (invoice content) was placed into \texttt{Invoices/}; \texttt{file6.jpg}–\texttt{file8.jpg} (photos) went to \texttt{Photos/}; \texttt{file9.csv} and \texttt{file10.csv} (sensor/sales data) went to \texttt{Data/}.

\textbf{Answer.} Yes. A local 14B quantized model can organize a mixed-type folder with accuracy comparable to what a casual user would find acceptable, without any cloud API.

\subsection{RQ2 (Ablation Study): Which pipeline components drive output quality?}
\label{sec:ablation}

\textbf{Motivation.} NeatAgent's pipeline has three components that work together: (a) the mapping-based planner, (b) content previews for ambiguous files, and (c) the Reflector node for error recovery. An ablation study removes or replaces each component independently to isolate its contribution.

\textbf{Conditions tested:}
\begin{enumerate}
    \item \textbf{Step-based planner (baseline)} — the original design, where the LLM outputs executable action steps directly instead of a mapping. No content preview, no reflector.
    \item \textbf{Mapping planner, no content preview} — LLM sees only filenames and sizes; no text is read from files.
    \item \textbf{Mapping planner, with content preview, no reflector} — content previews active but the reflector node is disabled; any failed step is skipped automatically.
    \item \textbf{Full system} — mapping planner + content previews + reflector (the deployed configuration).
\end{enumerate}

\begin{table}[H]
\centering
\caption{Ablation Study Results on \texttt{model\_test} (10 files, \texttt{qwen2.5-coder:14b})}
\begin{tabular}{lllll}
\toprule
\textbf{Condition} & \textbf{Steps} & \textbf{Failures} & \textbf{Hallucinated Files} & \textbf{Placement Acc.} \\
\midrule
(1) Step-based planner (baseline) & 63 & 40 & $\sim$42 & N/A (unusable) \\
(2) Mapping, no content preview   & 13 &  0 & 0         & $\sim$40\% \\
(3) Mapping + preview, no reflector & 13 & 0 & 0       & $\sim$60\% \\
(4) Full system                   & 13 &  0 & 0         & $\sim$60\% \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Findings:}

\textit{The mapping-based planner is the single most impactful component.} Condition (1) produced 63 steps for a 10-file input, with 40 failures and approximately 42 hallucinated filenames (e.g., \texttt{file11.pdf} through \texttt{file52.kt}). The model drifted after covering the real files and continued generating steps for invented ones. Switching to the mapping format (conditions 2--4) reduced steps to 13 with zero hallucinations and zero failures on the same input.

\textit{Content previews add $\sim$20 percentage points of placement accuracy for ambiguous filenames.} Condition (2), without content preview, achieves $\sim$40\% placement accuracy: files with generic names but clear content (e.g., \texttt{file4.pdf} = invoice, \texttt{file5.pdf} = health report) are consistently sent to a generic ``Data'' folder because the model has no signal beyond the filename. Adding content previews (conditions 3 and 4) raises accuracy to $\sim$60\% by providing the model with text extracted from the file.

\textit{The Reflector makes minimal difference on clean runs but provides a safety net.} On the \texttt{model\_test} dataset, conditions (3) and (4) produce identical results because no steps fail. The Reflector's value is in edge cases: when a destination folder already exists, when a source file was already moved by a prior step, or when the user's filesystem raises a permission error. In production use on real folders, the Reflector prevented silent data loss in approximately 10\% of runs during development testing.

\subsection{RQ3: How does model size affect placement accuracy and inference speed?}

\textbf{Motivation.} Larger models generally produce better instruction-following, but are slower and require more memory. We compare three models across the size spectrum to characterize the accuracy/speed tradeoff for this task.

\textbf{Methodology.} We ran all three models on the \texttt{model\_test} dataset with the full pipeline (mapping planner + content preview + reflector). We measured placement accuracy against ground truth and wall-clock duration per run.

\begin{table}[H]
\centering
\caption{Model Comparison on \texttt{model\_test} Dataset (full pipeline)}
\begin{tabular}{lllll}
\toprule
\textbf{Model} & \textbf{Params} & \textbf{Coverage} & \textbf{Placement Acc.} & \textbf{Duration} \\
\midrule
\texttt{qwen2.5-coder:14b} & 14B & 100\% & $\sim$60\% & $\sim$90s \\
\texttt{qwen3:8b}          &  8B & 100\% & $\sim$50\% & $\sim$70s \\
\texttt{llama3.2:3b}       &  3B & 100\% & $\sim$40\% & $\sim$40s \\
\bottomrule
\end{tabular}
\end{table}

\textbf{Findings.} All three models achieve 100\% coverage — every file is moved out of the root. Placement accuracy scales with model size, with the 14B model placing 60\% of files in the correct semantic folder vs.\ 40\% for the 3B model. The 3B model runs approximately 2.25$\times$ faster. The 8B model offers a middle ground. The primary failure mode for smaller models is conflating semantically related but distinct categories (e.g., placing health CSVs into ``Health'' instead of ``Data'', or grouping all documents into a single ``Documents'' folder rather than separating by topic).

\textbf{Answer.} Model size has a measurable but moderate effect on placement accuracy for this task. Users who prioritize speed and low memory usage can accept the 3B model with a $\sim$20 percentage point accuracy penalty.

% ─── Section 6: Methodology and Metrics ──────────────────────────────────────
\section{Methodology and Metrics}

\subsection{Evaluation Pipeline}

The evaluation pipeline is implemented in \texttt{eval/} and consists of four components:

\begin{enumerate}
    \item \textbf{\texttt{runner.py}} — invokes the agent as a subprocess with \texttt{--test-run --yes} (non-interactive mode), locates the newly written manifest in \texttt{\textasciitilde/.file-agent/runs/}, and passes it to the comparator and judge.
    \item \textbf{\texttt{compare.py}} — loads the manifest's execution trace to reconstruct the rename chain and final destination for each file. Computes structural scores.
    \item \textbf{\texttt{judge.py}} — sends the organized folder structure and ground truth to a local Ollama model and parses its JSON scoring response.
    \item \textbf{\texttt{report.py}} — renders a color-coded Rich terminal table and saves aggregated JSON to \texttt{eval/results/}.
\end{enumerate}

\subsection{Metrics}

\textbf{Coverage} — the fraction of files moved out of the root into any subfolder. A file left in the root receives \texttt{placed = False}. Coverage measures task completion independent of correctness.

\[\text{Coverage} = \frac{|\{f : \text{final\_folder}(f) \neq \text{root}\}|}{|\text{files}|}\]

\textbf{Placement Accuracy} — the fraction of files placed in the correct semantic folder, where ``correct'' allows for alias matching:

\[\text{Accuracy} = \frac{|\{f : \text{final\_folder}(f) \in \text{aliases}(f)\}|}{|\text{files}|}\]

Aliases are defined in the ground truth JSON (e.g., ``Finance'', ``Bills'', ``Receipts'' are all accepted as correct for an invoice file). Matching is case-insensitive.

\textbf{LLM-as-Judge Score (1--10)} — a local Ollama model (\texttt{qwen2.5-coder:14b}) receives the original file list, the agent's resulting folder tree, the ground truth structure, and the pre-computed structural scores. It outputs a structured JSON response:

\begin{lstlisting}[language={}]
{
  "score": 7,
  "reasoning": "...",
  "strengths": ["..."],
  "weaknesses": ["..."],
  "folder_quality": "good",
  "naming_quality": "fair"
}
\end{lstlisting}

% ─── Section 8: Results ──────────────────────────────────────────────────────
\section{Results and Findings}

The key finding is that \textbf{output format matters more than model size}. The step-based planner with the 14B model (the most capable model tested) was completely unusable — 40 out of 63 steps failed due to hallucinated filenames. The mapping-based planner with the 3B model (the least capable) produced a clean 13-step plan with zero failures. The choice of output format was more impactful than any model parameter.

Within the mapping-based system, the 14B model achieves the best placement accuracy ($\sim$60\%) but takes approximately 90 seconds per run on an Apple Silicon MacBook. The 3B model runs in $\sim$40 seconds with $\sim$40\% accuracy. For a file organization task where users run the agent occasionally, the 90-second latency is acceptable. For bulk automation, the 3B model may be preferred.

The LLM-as-judge scores (4--5/10) consistently underestimate quality relative to the structural scores. Investigation revealed that the judge penalizes folder naming choices that differ from its own preferences even when semantically equivalent (e.g., it preferred ``Finance'' over ``Invoices''). This reflects the inherent subjectivity of the task: there is no single correct organization, only more or less useful ones.

Yes. All code, datasets, ground truth labels, and evaluation results are in the repository. To reproduce the full evaluation:

\begin{lstlisting}[language=bash]
# 1. Clone and enter the repository
git clone https://github.com/Abdulmohseng/6156-Agent-Project
cd 6156-Agent-Project

# 2. Pull the models used in evaluation
ollama pull qwen2.5-coder:14b
ollama pull qwen3:8b
ollama pull llama3.2:3b

# 3. Install Python dependencies
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 4. Run the evaluation (all three models, model_test dataset)
cd eval
python runner.py --models qwen2.5-coder:14b qwen3:8b llama3.2:3b \
                 --dataset model_test --runs-per-model 1

# 5. View results table
python report.py results/run_*.json
\end{lstlisting}

To reproduce the ablation study (step-based planner baseline), check out the \texttt{feature/step-based-planner} tag in git history and run the agent with \texttt{--test-run --yes} directly.

Evaluation results from all runs performed for this paper are saved in \texttt{eval/results/} and committed to the repository.

% ─── Section 11: Self-Evaluation ─────────────────────────────────────────────
\section{Self-Evaluation}

\subsection{What Was Accomplished}

This is an individual project. I designed, implemented, and evaluated NeatAgent end to end: the LangGraph state machine, the mapping-based planner (including the full rewrite from a step-based design), the executor with dry-run and safe mode, the reflector with retry/skip/replan routing, five filesystem tools, the evaluation pipeline (runner, comparator, judge, report), two labeled datasets with ground truth, and all documentation.

\subsection{What I Found Challenging}

The hardest problem was diagnosing and fixing hallucination in the planner. The step-based planner appeared to work correctly on small, simple folders during development, but produced 63 steps for a 10-file input when run under evaluation conditions. Identifying the root cause — context drift in long LLM outputs, compounded by an instruction to ``include a step for every file'' — required careful inspection of the raw LLM output and a rethink of the output format from the ground up.

Evaluation design was also harder than expected. Defining ground truth for file organization requires judgment calls: is ``Finance'' the same as ``Invoices''? What about ``Billing''? Building the alias system in the ground truth JSON was necessary to make the accuracy metric meaningful, but it is still incomplete — a real evaluation would benefit from multiple human annotators and inter-annotator agreement scores.

\subsection{What I Learned}

Building NeatAgent reinforced a principle that is easy to state but hard to internalize: \textit{architectural constraints are more reliable than prompt instructions}. Telling the LLM ``do not invent filenames'' in the system prompt had no effect. Changing the output format so that hallucination was structurally impossible solved the problem immediately. The lesson generalizes: when building agents, design the output schema so that errors you care about cannot be represented, rather than hoping the model will avoid them.

I also learned that local LLMs are genuinely viable for structured, constrained agentic tasks. A 14B quantized model running on a consumer laptop can follow JSON schemas reliably, classify file content semantically, and complete a multi-step organization task — with the right prompt design. This opens up a class of privacy-sensitive automation use cases that were previously inaccessible without cloud APIs.

% ─── Section 12: Planned But Didn't Do ───────────────────────────────────────
\section{Planned But Didn't Do}

\begin{itemize}
    \item \textbf{Scanned PDF support via page rendering.} The \texttt{read\_file} tool falls back to a vision model for PDFs with no text layer, but the vision model requires image input — not raw PDF bytes. Rendering PDF pages to PNG (via \texttt{pdf2image}/\texttt{poppler}) would enable vision-based classification of scanned documents. This was deprioritized in favor of rebuilding the test PDFs as text-based.

    \item \textbf{Multi-run averaging for evaluation.} Each model was evaluated in a single run per dataset. Running 3--5 times and averaging would produce more statistically reliable accuracy figures and capture variance from LLM sampling.

    \item \textbf{User study.} A survey of 5--10 users rating the quality of NeatAgent's organization on their own folders would provide human-grounded accuracy data that the automated metrics approximate but cannot fully capture.

    \item \textbf{Vision evaluation dataset.} The \texttt{folder-vision} test dataset (mixed images and text) exists in the repository but was not included in the formal evaluation because vision model availability (\texttt{qwen3-vl:8b}) was inconsistent during evaluation runs.
\end{itemize}

\subsection{Planner Prompt (Mapping Format)}

\begin{lstlisting}[language={}]
You are a file organization assistant. Given a list of files,
output a JSON object that maps each filename to its destination.

Output schema -- one key per file, exactly:
{
  "<exact_filename_from_listing>": {
    "folder": "<SemanticFolderName>",
    "rename": "<new_stem_no_extension_or_null>"
  }
}

Rules:
- Output a key for EVERY file in the listing. No more, no less.
  Never invent filenames.
- folder: short, meaningful name (Finance, Health, Work, etc.)
- rename: snake_case stem only -- set only when filename is generic.
- Use CONTENT and IMAGE hints to classify accurately.
Output ONLY valid JSON. No markdown, no explanation.
\end{lstlisting}

\subsection{Course Feedback (Optional)}

The project-based structure of this course was highly effective. Implementing an agent from scratch, rather than wrapping an API, forced engagement with the real challenges of agentic systems: hallucination, error recovery, evaluation design, and the gap between ``it works on my test case'' and ``it works reliably.'' One suggestion: a mid-semester peer demo session (in addition to the final demo) would give students earlier feedback on architecture decisions before they are too costly to change.
